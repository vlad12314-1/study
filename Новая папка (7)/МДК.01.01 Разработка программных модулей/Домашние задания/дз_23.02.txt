1. Проблема "сырого" SQL
Если писать SQL-запросы прямо в коде интерфейса (GUI), возникают проблемы:

Код становится запутанным ("кашей").

Сложно тестировать логику работы с базой данных (БД).

Трудно менять тип СУБД (например, перейти с PostgreSQL на MySQL).

Много дублирования кода.

2. Решение: Data Access Layer (DAL)
DAL — это отдельный слой приложения, который отвечает за взаимодействие с базой данных. Он скрывает детали реализации SQL от остальных частей программы.

3. Основной паттерн: Repository (или DAO)

DAO (Data Access Object): Более низкоуровневый, 1 класс = 1 таблица.

Repository: Более высокоуровневый, работает с бизнес-сущностями, скрывает детали хранения.

Зачем нужен: Убирает SQL из GUI, уменьшает дублирование, упрощает поддержку кода.

Что делает Repository:

Выполняет SQL-запросы.

Обрабатывает ошибки БД.

Возвращает данные (в виде списков, словарей или DataFrame).

Что НЕ делает Repository: Не работает с GUI и не хранит состояние интерфейса.

4. Типичная структура

Подключение к БД: Создается в одном отдельном месте (функция get_connection).

Repository: Класс, содержащий методы для работы с конкретной сущностью (get_all(), add(), delete()).

GUI: Вызывает методы репозитория и получает готовые данные, ничего не зная об SQL.

5. Работа с Pandas
Репозиторий может возвращать данные не только в виде списков, но и в виде Pandas DataFrame, что удобно для дальнейшего анализа данных или отображения в таблицах.
